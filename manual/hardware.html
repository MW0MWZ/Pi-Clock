Hardware Guide

Supported Raspberry Pi models, display resolution limits, and performance considerations.

Supported Raspberry Pi Models

Pi-Clock runs on every Raspberry Pi model with HDMI output. The software automatically adapts its behaviour based on the number of CPU cores and available GPU capabilities.

Model CPU Cores RAM Max Resolution WiFi
Pi Zero W ARM11 @ 1 GHz 1 512 MB 1080p Built-in
Pi Zero 2 W Cortex-A53 @ 1 GHz 4 512 MB 1080p Built-in
Pi 1 Model B+ ARM11 @ 700 MHz 1 512 MB 1080p No (USB adapter)
Pi 2 Model B Cortex-A7 @ 900 MHz 4 1 GB 1080p No (USB adapter)
Pi 3 Model B/B+ Cortex-A53 @ 1.2–1.4 GHz 4 1 GB 1080p Built-in
Pi 4 Model B Cortex-A72 @ 1.5–1.8 GHz 4 1–8 GB 4K Built-in
Pi 5 Cortex-A76 @ 2.4 GHz 4 2–8 GB 4K Built-in

Recommended model: The Raspberry Pi Zero 2 W offers the best balance of cost, size, and performance for Pi-Clock. It has 4 cores (enabling smooth scroll, seconds display, and comfortable multi-layer rendering) in the same tiny form factor as the original Zero W. For 4K displays or heavy satellite tracking, choose a Pi 4 or Pi 5.

Display Resolution

Supported Resolutions

Resolution Pixels Supported Models
720p 1280 × 720 All models
1080p 1920 × 1080 All models (default)
1440p 2560 × 1440 Pi 4, Pi 5 only
4K 3840 × 2160 Pi 4, Pi 5 only (HDMI 2.0)

Single-core Pi models (Zero W, Pi 1) are capped at 1080p. Even if connected to a 4K monitor, the renderer will render at 1080p because the single-core CPU cannot sustain 4K rendering at an acceptable frame rate. The resolution cap is enforced automatically at boot.

Resolution Detection

Pi-Clock detects the display resolution in this priority order:

  1. User override — if you selected a specific resolution in the dashboard or pi-clock-renderer.conf, that resolution is used (requires reboot to apply).
  2. CPU core cap — single-core Pi models are capped at 1080p regardless of the display.
  3. Monitor EDID — the framebuffer auto-detects the native resolution from the monitor's HDMI EDID data.
  4. Fallback — if detection fails, defaults to 1080p.

Framebuffer Requirements

The renderer requires a 32-bit colour depth (ARGB8888) framebuffer. This is set automatically in the Pi-Clock OS image via framebuffer_depth=32 in /boot/firmware/config.txt. If you are running Pi-Clock on a custom Alpine Linux installation, ensure this setting is present.

Single-Core vs Multi-Core Behaviour

Pi-Clock automatically detects the number of CPU cores at startup and adapts several features accordingly. This ensures the best possible experience on every Pi model without manual tuning.

Feature Single-Core (Pi Zero W, Pi 1) Multi-Core (Pi 2, 3, 4, 5, Zero 2 W)
Clock display Hours and minutes only Hours, minutes, and seconds
HUD update rate Every 60 seconds Every 1 second
News ticker mode Headlines only (fly-in, hold, next) Smooth scroll or Headlines (user choice)
Ticker frame rate 30 FPS Display refresh rate
Max resolution 1080p (hard cap) Up to 4K (Pi 4/5)
Satellite tracking 3–5 satellites recommended All satellites comfortably

All other features (layers, applets, DX cluster, solar weather, propagation models) work identically on every model. The renderer's layer-caching system means static layers like grids and borders have zero ongoing CPU cost regardless of hardware.

Performance Considerations

Layer CPU Cost

Not all layers are equal in CPU usage. Here's a rough guide to help you choose which layers to enable on constrained hardware:

CPU CostLayersNotes
Zero Base Map, Borders, Grid, Time Zones, CQ Zones, Maidenhead Static layers — rendered once, composited from cache. No ongoing cost.
Very Low Sun, Moon, QTH Marker, HUD (60s mode) Simple calculations, update only every 60 seconds.
Low HUD (1s mode), DX Spots, News Ticker (headlines), Lightning, Earthquakes Frequent updates but lightweight rendering. Lightning and earthquake layers re-render every second but draw very few primitives.
Zero (cached) Rain Radar, Cloud Cover, Wind, Aurora Rendered once when new data arrives (every 6h for GFS, 15min for aurora), then cached on a surface. Zero per-frame cost. Wind streamlines take ~2-3s to compute on Pi Zero W but this happens only every few hours.
Medium Daylight, News Ticker (smooth scroll), Satellite Tracking Daylight does per-pixel solar math every 60s. Satellites propagate on every frame. Smooth scroll animates continuously.
Higher Propagation Heat Map Computes MUF/LUF for 180×90 grid cells. Recalculates every minute.

Recommendations by Model

Pi Zero W (1 core, 512 MB)

The most constrained platform. Pi-Clock is specifically optimised for this hardware, but be selective about which features you enable.

  • Stick to 1080p or 720p
  • The core display (map + daylight + clock + borders) runs comfortably
  • Add DX spots, solar weather applet, and a few layers without issue
  • Limit satellite tracking to 3–5 satellites
  • Use headlines mode for the news ticker (forced automatically)
  • The propagation heat map works but adds noticeable load
  • Avoid enabling everything simultaneously

Pi Zero 2 W / Pi 2 / Pi 3 (4 cores, 512 MB–1 GB)

A significant step up from the single-core models. Comfortable for most configurations.

  • 1080p renders very smoothly
  • All layers and applets can run simultaneously
  • Smooth scroll ticker works well
  • Track 10+ satellites without issue
  • Propagation heat map + VOACAP applet together are fine

Pi 4 / Pi 5 (4 cores, 1–8 GB)

More than enough power for everything Pi-Clock offers.

  • 4K rendering works well (Pi 4 may drop frames with all layers; Pi 5 is smooth)
  • Enable everything — all layers, all applets, all satellites
  • Dual HDMI outputs on Pi 4/5 (Pi-Clock uses the first HDMI port)
  • Consider 4K for wall-mounted displays where detail matters

Display Hardware

HDMI adapters: Pi Zero models need a mini-HDMI to HDMI adapter or cable. Pi 4/5 need micro-HDMI to HDMI. Make sure you have the right adapter for your Pi model.

Why 30 Hz at 4K? Pi-Clock always uses 4K @ 30 Hz rather than 60 Hz. This ensures maximum compatibility across display hardware and avoids a known issue on the Pi 5 where 4K @ 60 Hz causes interference with the 2.4 GHz WiFi radio.

Recommended Displays

Pi-Clock works with any HDMI display. For the best experience:

  • Size: 24” or larger for comfortable viewing from across a room or shack
  • Resolution: 1080p is the sweet spot for most setups. 4K is beautiful but requires a Pi 4/5
  • Panel type: IPS panels offer better viewing angles for a wall-mounted clock
  • TV vs monitor: Either works. TVs often have a "PC mode" or "game mode" that reduces input lag and overscan

Overscan: Some TVs apply overscan (cropping the edges of the image). If the clock text or applets are cut off at the screen edges, look for an "overscan", "fit to screen", or "1:1 pixel mapping" option in your TV's settings.

Display Output Method

Pi-Clock renders directly to the Linux framebuffer (/dev/fb0) via SDL2 with the KMSDRM backend. No X11 window manager, no Wayland compositor, and no desktop environment is needed. This gives the renderer direct access to the display hardware with minimal overhead.

Pi ModelDisplay DriverGPU
Pi Zero / Pi 1 / Pi 2 / Pi 3Legacy framebuffer (/dev/fb0)VideoCore IV
Pi 4KMS/DRM (vc4-kms-v3d-pi4)VideoCore VI
Pi 5KMS/DRM (vc4-kms-v3d-pi5)VideoCore VII

The renderer automatically selects the correct driver for your Pi model.

Power Supply

Use the correct power supply for your Pi model to avoid under-voltage issues (which cause CPU throttling and display glitches):

Pi ModelConnectorRecommended
Pi Zero / Zero W / Zero 2 WMicro USB5V 2.5A
Pi 1 / Pi 2 / Pi 3Micro USB5V 2.5A
Pi 4USB-C5V 3A (official PSU recommended)
Pi 5USB-C5V 5A (official 27W PSU recommended)

Under-voltage: If the Pi is under-powered, the CPU throttles and the display may stutter or freeze. The renderer logs a warning if it detects CPU throttling. Use the official Raspberry Pi power supply for best results.

SD Card

  • Minimum size: 4 GB (the OS image with both slots and data partition fits in approximately 3.5 GB)
  • Speed class: Class 10 / UHS-I or faster. Affects boot time, not runtime performance.
  • Endurance: For 24/7 operation, use a high-endurance card (Samsung PRO Endurance, SanDisk MAX Endurance). Pi-Clock minimises writes, but SD cards do degrade over time.

News tickerYes (RSS feeds)
Web dashboard accessYes (local network)

WiFi vs Ethernet: WiFi is sufficient for all Pi-Clock data needs (the data rates are very low — a few kilobytes every few minutes). Ethernet is more reliable for 24/7 operation and avoids WiFi dropout issues.

Thermal Considerations

Pi-Clock's CPU usage is moderate (it's not constantly rendering at maximum frame rate), but for 24/7 operation:

  • Pi Zero / Zero W: Typically fine without a heatsink. The single-core adaptations keep CPU usage low.
  • Pi Zero 2 W / Pi 3: A small heatsink is recommended for sustained operation, especially if many layers are enabled.
  • Pi 4 / Pi 5: A heatsink or case with passive cooling is recommended. The official Pi 4/5 cases with built-in heatsinks work well. Active cooling is unnecessary for Pi-Clock.

The OS sets a thermal throttle limit of 75°C in config.txt. If the CPU reaches this temperature, it will reduce clock speed to prevent overheating.
