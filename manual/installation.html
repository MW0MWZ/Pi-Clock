Installation & OS Architecture

How Pi-Clock OS is built, how it boots, how it upgrades, and how your configuration persists.

Installing from an SD Card Image

The recommended way to install Pi-Clock is to flash the pre-built OS image to an SD card. The image contains the complete operating system, renderer, dashboard, and maps.

Requirements

  • A micro SD card (4 GB minimum)
  • A computer with an SD card reader
  • An imaging tool: Raspberry Pi Imager, balenaEtcher, or dd

Flashing the Image

Download the latest .img.xz file from the Pi-Clock home page and write it to the SD card using Raspberry Pi Imager. Select "Use custom" in the OS menu and choose the downloaded file. After the write completes, re-insert the card and copy your pi-clock-config.txt to the BOOT drive.

Full step-by-step imaging guide with screenshots →

Alternatively, use dd on Linux/macOS:

xzcat pi-clock-os-*.img.xz | sudo dd of=/dev/sdX bs=4M status=progress
sync

Installing via APK Packages

If you already have an Alpine Linux system (or RasPINE), you can install Pi-Clock as individual APK packages without reflashing the SD card.

1

Add the signing key

wget -O /etc/apk/keys/pi-clock-apk-signing.rsa.pub \
    https://pi-clock.pistar.uk/pi-clock-apk-signing.rsa.pub
2

Add the repository

echo "https://pi-clock.pistar.uk/v3.21/pi-clock" \
    >> /etc/apk/repositories
3

Install the packages

apk update
apk add pi-clock-maps pi-clock-renderer pi-clock-dashboard

Operating System Architecture

Pi-Clock runs on RasPINE — a custom OS based on the Raspberry Pi OS kernel and modules combined with an Alpine Linux userland (musl libc, apk package manager). This gives the best hardware support (Raspberry Pi Foundation's kernel) with the smallest possible footprint (Alpine is typically 50–100 MB for a full system).

SD Card Layout

The SD card is divided into four partitions:

P1: /boot/firmware 256 MB FAT32 Kernel, firmware, config files P2: Slot A 1.5 GB ext4 Root filesystem (active OR inactive) P3: Slot B 1.5 GB ext4 Root filesystem (active OR inactive) P4: /data grows ext4 Persistent data (auto-expands first boot)
Partition Mount Point Purpose
P1 (boot) /boot/firmware FAT32 — kernel, device tree files, firmware, config.txt, cmdline.txt, slot.conf, and pi-clock-config.txt. This partition is accessible from any computer (just insert the SD card).
P2 (Slot A) / (when active) ext4 — complete root filesystem. One of the two A/B upgrade slots.
P3 (Slot B) / (when active) ext4 — complete root filesystem. The other A/B upgrade slot.
P4 (data) /data ext4 — persistent data that survives OS upgrades. Configuration files, SSH keys, WiFi credentials, and user home directory. Auto-expands to fill the SD card on first boot.

Boot Process

The boot sequence runs through three OpenRC runlevels:

1. sysinit (Immediate)

  • pi-clock-splash — displays the Pi-Clock logo directly to the framebuffer, suppresses kernel messages, and switches to a dedicated virtual terminal. This ensures the display shows something immediately rather than a blank screen or scrolling boot text.

2. boot (System Setup)

  • pi-clock-data-resize — on first boot only, expands the /data partition (P4) to fill the entire SD card. This is a two-phase process: the partition table is updated first (requires a reboot), then the filesystem is grown to match. The service disables itself after completion.
  • pi-clock-bindmounts — bind-mounts directories from /data/etc/ over their corresponding /etc/ locations. This is how configuration persists across OS upgrades — the rootfs is replaced, but /data is untouched.
  • pi-clock-config — processes the zero-touch configuration file on first boot, or applies saved settings from /data/etc/pi-clock-user.conf on subsequent boots.
  • pi-clock-boot-watchdog — if an upgrade is pending (PENDING=true in slot.conf), increments the boot attempt counter. After three consecutive failed boots, automatically rolls back to the previous OS slot.
  • Standard services: networking, DHCP, wpa_supplicant (WiFi), Dropbear (SSH)

3. default (Application Services)

  • pi-clock-health-check — waits 30 seconds, then checks if the renderer is running or the network is up. If either check passes, the upgrade is marked as successful (PENDING cleared). If both fail, the slot remains pending and will be rolled back after three total failed boots.
  • pi-clock-renderer — detects the display resolution (from the framebuffer or user override) and starts the full-screen map renderer.
  • pi-clock-dashboard — starts the web management UI on HTTPS port 443.

Boot times: First boot takes approximately 60 seconds due to the partition resize and reboot. Subsequent boots take 20–35 seconds depending on your Pi model. The splash screen is visible almost immediately.

A/B Slot Upgrades

Pi-Clock uses an A/B slot upgrade system inspired by Android and ChromeOS. The two root filesystem partitions (Slot A and Slot B) alternate roles: one is the active system, the other holds the previous version or receives the next upgrade.

How an Upgrade Works

1

Download

A new rootfs image is downloaded to the inactive slot. The running system is untouched during this process.

2

Verify

The downloaded image is verified against a SHA256 checksum before installation.

3

Install

The inactive slot is formatted and the new rootfs is extracted. The boot configuration is updated to point to the new slot, and PENDING is set to true.

4

Reboot

The system reboots into the new slot. The boot watchdog begins counting attempts.

5

Validate

After 30 seconds, the health check confirms the renderer is running or the network is up. If successful, PENDING is cleared and the upgrade is complete.

Automatic Rollback

If the new slot fails to boot successfully three times in a row (the renderer is not running and the network is not up), the boot watchdog automatically rolls back to the previous slot and reboots. This ensures that a bad upgrade can never permanently brick your Pi-Clock.

Safe upgrades: Your configuration, WiFi credentials, SSH keys, and all settings are stored on the /data partition and are completely unaffected by rootfs upgrades. You never need to reconfigure after an upgrade.

Manual Rollback

You can manually roll back to the previous slot from the dashboard (System page) or via SSH:

pi-clock-rollback        # Switch boot to previous slot (requires manual reboot)
pi-clock-rollback --force # Switch and reboot immediately

Persistence Across Upgrades

The /data partition (P4) survives all OS upgrades. Pi-Clock uses bind mounts to overlay persistent configuration from /data/etc/ over the rootfs /etc/ directories at boot time.

What Persists

Data Location
All Pi-Clock config /data/etc/pi-clock-*.conf
WiFi credentials /data/etc/wpa_supplicant/
SSH host keys /data/etc/dropbear/
User home directory /data/home/pi-clock/
User password & hostname /data/etc/pi-clock-user.conf

What Does NOT Persist

  • /tmp — mounted as tmpfs (RAM), cleared on every reboot
  • /var/log — mounted as tmpfs, logs are lost on reboot (reduces SD card wear)
  • Anything installed manually on the rootfs (P2/P3) — replaced on upgrade

Zero-Touch Configuration

You can pre-configure Pi-Clock before first boot by placing a file called pi-clock-config.txt on the boot partition (P1, FAT32). This is the same partition visible when you insert the SD card into any computer.

The file uses a simple key=value format, one setting per line. Comments start with #.

Available Settings

Key Description Example
wifi_ssid WiFi network name MyNetwork
wifi_password WiFi password (WPA2, 8–63 characters) MySecretKey
wifi_country WiFi regulatory domain (2-letter ISO code) GB, US, DE
hostname System hostname shack-clock
user_password Set the dashboard login password MyPassword123
timezone Timezone (Olson format) Europe/London
callsign Your amateur radio callsign M1ABC
qth_lat QTH latitude (-90 to 90) 51.5
qth_lon QTH longitude (-180 to 180) -0.1
grid_square Maidenhead grid square IO91wm
center_lon Map center longitude (-180 to 180) 0.0
display_resolution Display resolution 1080p, 4k, native
enable_ssh_password Allow SSH password login (default: key-only) true
ssh_key SSH public key to authorize ssh-ed25519 AAAA...

Multiple WiFi Networks

You can configure up to 10 WiFi networks with numbered suffixes. Higher-numbered networks are tried first (priority):

wifi_ssid_1=HomeNetwork
wifi_password_1=HomePassword
wifi_ssid_2=ShackNetwork
wifi_password_2=ShackPassword

Security: The pi-clock-config.txt file is automatically deleted after processing because it may contain passwords in plain text. The password is hashed and stored securely in /data/etc/pi-clock-user.conf.

SSH Access

Pi-Clock runs Dropbear (a lightweight SSH server) on port 22. By default, only key-based authentication is enabled for security.

Connecting via SSH

ssh pi-clock@pi-clock.local

Enabling Password Authentication

If you need password-based SSH access (not recommended for production), set enable_ssh_password=true in pi-clock-config.txt before first boot, or change it via the dashboard.

Adding SSH Keys

Add your SSH public key via pi-clock-config.txt:

ssh_key=ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIExample... user@host

Or copy it manually after first boot:

ssh-copy-id pi-clock@pi-clock.local

SD Card Longevity

Pi-Clock is designed to minimise SD card writes:

  • /tmp and /var/log are mounted as tmpfs (RAM) — no disk writes for temporary files or logs
  • Configuration changes are written atomically (temp file, sync, rename) to prevent corruption on power loss
  • The renderer only writes to disk when configuration changes, not during normal operation
  • Use a high-endurance SD card (Samsung PRO Endurance or similar) for best results
